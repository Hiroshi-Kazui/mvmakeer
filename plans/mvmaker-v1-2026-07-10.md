## Requirements

`C:\develop\mvmaker\要件定義.md`(2026-07-10版)の実装。要点:

- 画像(jpg/png)+ 音声(mp3/wav/m4a)からスライドショーMVを作成するデスクトップツール
- プレビューはブラウザ側リアルタイム合成(中間動画なし)。ffmpegは最終書き出しのみ
- 再生しながら `I` でテロップ挿入(挿入後にテキスト入力)、`O` でフェードアウト位置設定
- テロップ: フォント/サイズ/色/9分割位置、フェードイン0.5s・アウト0.8s(個別変更可)、同時表示は1つ(重なりは自動で詰める)
- タイムライン: テロップ/画像区間/音声波形、ドラッグ編集
- 画像表示時間: 均等割り初期値 → 個別編集(残時間再配分)
- クロスフェード固定(1.0s)、1920×1080/30fps固定、mp4(H.264+AAC)焼き込み出力
- プロジェクト保存/再開(`.mvproj` = JSON、素材は絶対パス参照)
- スタック: Tauri 2 + TypeScript + Vite(素のTS)、ffmpegサイドカー。`週日の集会割当` と同構成

## 技術判断(プラン時点で決定するもの)

1. **ディレクトリ構成**: 参照プロジェクトを踏襲 — `mvmaker/app`(フロント+src-tauri)、ルートに `run.cmd`
2. **ffmpeg サイドカー**: gyan.dev の Windows full ビルド(libass入り、GPL)を `src-tauri/binaries/ffmpeg-x86_64-pc-windows-msvc.exe` に配置し `bundle.externalBin` 登録。実行は `tauri-plugin-shell` の sidecar 経由。※GPLビルドのため、将来配布時はライセンス表記 or LGPL構成へ差し替えを再検討(要件§8どおり先送り)
3. **素材のWebView読込**: Tauri の asset protocol(`convertFileSrc`)を使用。`app.security.assetProtocol.enable=true` + scope 設定
4. **テロップ焼き込み**: 第一候補 **ASS字幕**(`\fad` でフェード、`\an1-9` で9分割位置、Style行でフォント/サイズ/色)。Phase 1 PoC で日本語フォント解決を検証し、不成立なら drawtext に切替
5. **波形**: Web Audio API `decodeAudioData` でピーク抽出 → canvas 描画。m4a がデコード不可の場合は ffmpeg サイドカーで PCM 抽出にフォールバック
6. **プレビュークロック**: `<audio>.currentTime` を唯一の時刻源とし、rAF で描画同期(±0.1s 要件はこれで満たす)
7. **クロスフェード書き出し**: `xfade` フィルタのチェーン。画像50枚でフィルタグラフが破綻する場合はセグメント分割エンコード+concat にフォールバック(PoCで判定)

## Implementation Phases

### Phase 0: スキャフォールド(基盤)
- Steps:
  1. `app/` に Tauri 2 + Vite + vanilla-ts プロジェクト作成(参照プロジェクトの設定を雛形に)
  2. 依存追加: `@tauri-apps/plugin-dialog`, `@tauri-apps/plugin-shell`(sidecar用), `@tauri-apps/plugin-fs`(保存用)
  3. ffmpeg full ビルドをダウンロードし sidecar 配置、`externalBin`・capabilities 設定
  4. `run.cmd` 作成
- 動作確認: `npm run tauri dev` でウィンドウが開き、アプリ内から sidecar `ffmpeg -version` の出力が取れる
- Dependencies: なし
- Risk: LOW

### Phase 1: ffmpeg 書き出し PoC(最大リスクの先行検証)
- Steps:
  1. `poc/` にサンプル素材(画像3枚+短い音声)を用意し、手動コマンドで検証:
     a. xfade クロスフェード合成 + 音声多重化
     b. ASS 字幕(日本語・游明朝・`\fad`・`\an` 位置指定)の焼き込み
  2. 画像50枚相当の xfade チェーンを生成しエンコード時間・安定性を実測 → 直列 or セグメント分割を決定
  3. ass vs drawtext の最終決定、フォント指定方法(fontsdir 等)を確定
  4. 結果と採用コマンド形を `poc/RESULT.md` に記録
- 動作確認: テロップ入りmp4が正しく再生でき、フェード・位置・日本語表示が仕様どおり
- Dependencies: Phase 0(sidecar)
- Risk: HIGH(このフェーズで方式確定させ、以降の手戻りを防ぐ)

### Phase 2: ドメインモデルと永続化
- Steps:
  1. `types.ts` — Project / ImageEntry / Telop / TelopStyle 型(.mvproj 構造 = 要件§4.6)
  2. `logic/timing.ts` — 均等割り、個別秒数変更時の残時間再配分、テロップ重なり自動詰め、正規化(tIn < tFo ≤ tEnd)
  3. `logic/projectFile.ts` — 保存/読込(dialog + fs)、素材欠落検出と再指定フロー
  4. `state.ts` — アプリ状態と変更通知(dirty フラグ含む)
- 動作確認: ロジックの単体テスト(vitest)が通る。保存→読込のラウンドトリップ一致
- Dependencies: Phase 0
- Risk: LOW

### Phase 3: UIシェルと素材管理
- Steps:
  1. モックの HTML/CSS を `index.html` + `src/ui/` に移植(歌詞キューは除去)
  2. 画像追加(複数選択ダイアログ)→ `convertFileSrc` でサムネイル表示、D&D並べ替え、削除
  3. 音声選択 → duration 取得 → 均等割り反映
- 動作確認: 実ファイルで素材一覧・並べ替え・差し替えが機能する
- Dependencies: Phase 2
- Risk: LOW

### Phase 4: プレビューエンジン
- Steps:
  1. `preview/engine.ts` — audio.currentTime 駆動の描画ループ、再生/停止/シーク
  2. 画像切替+クロスフェード描画(書き出しと同じ1.0s・同じカーブになるようパラメータ共有)
  3. テロップオーバーレイ(フェード不透明度、スタイル、9分割位置)
  4. トランスポートUI接続(Space/←→/シークバー/タイムコード)
- 動作確認: 実素材でプレビュー再生でき、シークしても画像・テロップ表示が正しい
- Dependencies: Phase 3
- Risk: MEDIUM(タイミング精度・描画負荷)

### Phase 5: テロップ打ち込みワークフロー
- Steps:
  1. `I` 挿入 → アクティブ化 → 右パネルのテキスト欄へフォーカス(再生は継続)
  2. `O` でフェードアウト位置設定、`Del` 削除(確認ダイアログ)
  3. インスペクタ: テキスト/フォント(システムフォント列挙)/サイズ/色/位置/タイミングステッパー
  4. 同時表示1つの自動詰めを挿入時に適用
- 動作確認: 曲を流しながら10個打ち込み→修正、の一連が要件どおり動く
- Dependencies: Phase 4
- Risk: MEDIUM

### Phase 6: タイムライン
- Steps:
  1. ルーラー/プレイヘッド/シーク(モックの実装を移植・拡張)
  2. テロップクリップ: 選択・移動・両端/Oマーカードラッグ(重なり詰めと整合)
  3. 画像トラック: 境界ドラッグで表示時間編集(残時間再配分)
  4. `preview/waveform.ts` — デコード→ピーク→canvas。m4a 不可時は ffmpeg フォールバック
- 動作確認: ドラッグ編集がプレビューへ即時反映、波形と音が視覚的に一致
- Dependencies: Phase 4, 5(Phase 2 のロジックを使用)
- Risk: MEDIUM

### Phase 7: 書き出し
- Steps:
  1. `export/assBuilder.ts` — テロップ配列→ASS(PoCで確定した書式)
  2. `export/ffmpegCommand.ts` — プロジェクト→filter_complex(PoCで確定した構成)
  3. Rust側: sidecar 実行、`-progress` パース→進捗イベント、キャンセル(kill)
  4. 書き出しUI: 保存先選択、進捗バー、キャンセル、完了後フォルダを開く
- 動作確認: 実プロジェクトを書き出し、プレビューと見た目・タイミングが一致(目視+数点のタイムコード比較)
- Dependencies: Phase 1(方式), 2, 5
- Risk: MEDIUM(PoC済みのため HIGH→MEDIUM に低減)

### Phase 8: 仕上げ
- Steps:
  1. 未保存終了ガード、破壊的操作の確認ダイアログ総点検
  2. エラー表示のユーザー向け整備(sidecar失敗、素材欠落等)
  3. 性能確認: 画像50枚/テロップ100個/10分曲で操作快適性を実測
  4. README(セットアップ・ffmpeg配置手順)整備
- 動作確認: 要件§5の非機能要件を満たす
- Dependencies: Phase 3–7
- Risk: LOW

## Risks

- HIGH: xfade 50枚チェーンの実用性(グラフ肥大・メモリ・時間)→ Phase 1 で実測し、セグメント分割+concat へのフォールバックを用意
- HIGH: ASS の日本語フォント解決(Windows の fontconfig 挙動)→ Phase 1 で検証、`fontsdir` 指定 or drawtext 切替
- MEDIUM: プレビューと書き出しの見た目・タイミング差(クロスフェードカーブ、カバークロップ計算)→ パラメータを単一ソース化し、Phase 7 で照合
- MEDIUM: WebView2 の `decodeAudioData` が m4a 非対応の可能性 → ffmpeg で PCM 抽出フォールバック(Phase 6)
- MEDIUM: ffmpeg full ビルドは GPL → 個人利用は問題なし。配布時に再検討(要件§8)
- LOW: 長尺曲の波形データのメモリ量 → ピークをダウンサンプリング
